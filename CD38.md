# CD38 当前能力总览

这个文件回答一个核心问题：当前项目针对 CD38 已经能做到什么程度、结果可信边界在哪里、下一步最短该补什么。

## 一页结论

目前 CD38 模块已经可以作为“口袋识别准确性检查 + 外部 pocket 工具接入 + 比赛展示用 benchmark”的专项流程使用。它已经能用公开 CD38 结构，把预测口袋和已知 CD38 关键活性位点做覆盖率、精确率、F1、近邻覆盖和过宽风险评估。

当前最明确的结果是：在已有 4 条 benchmark 记录中，`3ROP` 和 `4OGW` 的 ligand-contact / P2Rank baseline 都能覆盖大部分或全部 CD38 compact truth residues。P2Rank 在当前两个结构上 exact truth coverage 都是 `1.0`，说明核心活性位点能被找到；但 `4OGW_p2rank_rank1_chainA` 的预测范围偏宽，precision 只有 `0.2692`，overwide pocket score 达到 `0.6175`，所以它更适合作为“找得到核心区域，但需要过滤过宽口袋”的例子。

当前不能把它包装成“CD38 口袋预测软件已经完整验证”。原因是还缺真实 `fpocket` 输出，`3F6Y` 的 P2Rank/fpocket 结果还没补齐，结构数量也太少，不足以改变全局 scoring 默认参数。

### 3. ligand-contact baseline

当前已经能从带 ligand 的 CD38 结构中，用 ligand 附近残基作为简化 pocket baseline。

| 结果名 | 预测残基数 | truth 覆盖 | precision | F1 | 近邻 truth 覆盖 |
|---|---:|---:|---:|---:|---:|
| `3ROP_ligand_contact_chainA_50A_NCA` | 13 | 0.8571 | 0.4615 | 0.6000 | 1.0000 |
| `4OGW_ligand_contact_chainA_NMN` | 14 | 1.0000 | 0.5000 | 0.6667 | 1.0000 |

这说明 ligand-contact baseline 对 CD38 的核心活性区域有实用价值，尤其适合作为 P2Rank/fpocket 的 sanity check。

### 4. P2Rank benchmark

当前已经导入并评估了两个 P2Rank baseline：

| 结果名 | P2Rank rank | 预测残基数 | truth 覆盖 | precision | F1 | overwide score |
|---|---:|---:|---:|---:|---:|---:|
| `3ROP_p2rank_rank2_chainA` | 2 | 14 | 1.0000 | 0.5000 | 0.6667 | 0.2917 |
| `4OGW_p2rank_rank1_chainA` | 1 | 26 | 1.0000 | 0.2692 | 0.4242 | 0.6175 |

结论：P2Rank 对 CD38 核心口袋定位是有效的，但不同结构上可能产生过宽预测。当前系统已经能把“找得到”和“预测过宽”分开评价，而不是只看是否覆盖 truth。

### 5. 参数敏感性和几何 proxy 校准

当前 CD38 benchmark 已经能生成参数敏感性、proxy calibration 和过宽风险报告。最近一次 proxy calibration 的推荐是：

| 参数 | 当前建议 |
|---|---|
| runtime proxy threshold | `0.55` |
| 默认 overwide penalty weight | `0.0` |
| 实验性 penalty weight | `0.15` |
| 当前策略 | `keep_default_penalty_off` |
| 证据等级 | `low` |

解释：当前 CD38 数据能说明“过宽风险需要被展示和审计”，但还不足以把 overwide penalty 变成全局默认扣分项。更稳妥的做法是继续保留报告和 sensitivity review，而不是直接改变模型默认分数。

### 7. 和主 ML 流程的关系

CD38 专项 benchmark 主要验证的是“pocket/活性位点定义和预测口袋质量”。如果你提供的是 CD38-抗体或 CD38-VHH pose 表，并且输入里包含 pocket/catalytic residues，主 ML 流程可以继续做：

| 能力 | 是否已有 |
|---|---|
| 读取 CD38 pocket/catalytic residue 定义 | 已有 |
| 支持 `A:37-40`、`B:37-40`、`C:37-40` 这类范围写法 | 已有 |
| 计算 pocket blocking / mouth / path / contact proxy | 已有 |
| 规则分数 + MLP 分数 + consensus ranking | 已有 |
| 候选解释卡、质量门控、实验建议 | 已有 |
| 基于 CD38 催化/功能残基的 catalytic-anchor 3D shell pocket 诊断 | 已有 |
| 项目级 pocket evidence：从 `A/rsite/result` 自动选代表 PDB 并回填 `candidate_curated_pocket.txt` | 已有 |
| P2Rank/fpocket 过宽预测的 precision guard | 已有 |
| AI prior 离线审计，防止 AI 抽取直接当 ground truth | 已有 |
| CD38 专属 VHH wet-lab 标签验证 | 尚未有 |

所以当前能做的是“CD38 口袋定义可信度检查 + CD38 相关 pose 的阻断候选排序”，还不能声称“已经完成 CD38 抗体药物发现验证”。

补充：主 ML 现在已经支持 catalytic-anchor pocket 诊断。如果把 CD38 文献或 benchmark truth 中的催化/功能残基写进 `catalytic_file`，系统会在每个 PDB pose 中围绕这些残基生成 4A、6A、8A shell，并输出 `catalytic_anchor_*` 列。这个能力适合处理“文献有关键催化残基，但没有直接 pocket 位置注释”的情况；它用于解释和人工复核，不替代外部 pocket finder benchmark。

新增：现在还可以先运行 `build_pocket_evidence.py` 做 CD38 pocket 证据整合。它会把 `rsite/rsite.txt` 派生出的人工 pocket、文献/催化残基、ligand-contact、P2Rank、fpocket 和 AI prior 合并到 `pocket_evidence.csv`，再输出 `candidate_curated_pocket.txt` 和 `review_residues.txt`。在当前 A/CD38 样例中，用同一份 rsite 派生 pocket 同时作为 `manual_pocket_file` 和 `catalytic_file` 时，脚本能生成 23 个 curated candidate，并正确提示 catalytic anchor shell 偏宽，说明它不会把“宽 pocket 文件”误当成精确催化核心。

补充：项目级入口 `build_project_pocket_evidence.py` 已在当前 A/CD38 目录上跑通。它能自动发现 `A/rsite/result`，按 `MMPBSA_energy` 最低值选择代表 PDB，生成项目级 pocket evidence，并写出带 `candidate_curated_pocket.txt` 的 `input_pose_table_with_pocket_evidence.csv`。该输入表继续跑 `build_feature_table.py` 时，当前 180 行全部成功。

当前 pocket evidence 的安全策略：

- 如果 P2Rank/fpocket 给出过宽 pocket，缺少 manual/literature/catalytic core/ligand-contact 高置信支持的 residue 会标记为 `external_overwide_guard`，只进入 `review_residues.txt`。
- 如果提供 AI prior，它只作为待复核线索，不参与 curated 判定的支持分/方法数；AI 来源必须通过 `ai_prior_audit.csv` 保留来源句子、证据等级和人工复核状态。

示例：

```bash
python build_pocket_evidence.py ^
  --pdb_path A\rsite\result\vhh1\CD38_1\1\1.pdb ^
  --manual_pocket_file A\cd38_pocket_from_rsite_chain_B.txt ^
  --catalytic_file A\cd38_pocket_from_rsite_chain_B.txt ^
  --antigen_chain B ^
  --out_dir smoke_test_outputs\pocket_evidence_check
```

项目级最短命令：

```bash
python build_project_pocket_evidence.py ^
  --project_root A ^
  --target_prefix CD38
```

## 下一步最短路径

如果只想刷新当前 CD38 public starter：

```powershell
python run_cd38_public_starter.py
```

如果要补齐外部工具结果，推荐流程是：

```powershell
python package_cd38_external_tool_inputs.py
```

然后把生成的 transfer package 放到装有 P2Rank/fpocket 的 WSL、Linux 或外部机器上运行。拿回返回包后执行：

```powershell
python finalize_cd38_external_benchmark.py --import_source <returned_zip_or_dir> --run_discovered --run_sensitivity
```

这一步完成后，最应该观察的是：

| 观察项 | 看什么 |
|---|---|
| fpocket 是否补齐 | `3ROP`、`4OGW`、`3F6Y` 是否出现 fpocket 行 |
| 3F6Y 是否补齐 | 是否出现 `3F6Y_p2rank` 和 `3F6Y_fpocket` 相关行 |
| P2Rank/fpocket 是否一致 | 是否都覆盖 A:125/A:127/A:146/A:155/A:189/A:193/A:226 附近 |
| 是否过宽 | precision、far predicted fraction、overwide pocket score 是否异常 |
| 默认参数是否能调整 | proxy calibration evidence level 是否从 low 提升 |
